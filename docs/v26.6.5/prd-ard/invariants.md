# System Rules & Invariant Queries: lsp-max v26.6.5

This document details the five core system invariants of the `lsp-max` Admitted Graph Control Plane and defines the SPARQL 1.1/1.2 validation queries used to enforce them.

---

## 1. Overview of System Invariants

To guarantee functional correctness, metadata provenance, and cryptographic determinism, the Admitted Graph Control Plane operates under five strict invariants. If any invariant is violated, the control plane blocks SNAPSHOT updates and refuses to promote the release status to ALIVE.

---

## 2. Invariant 1 — No Orphan LSIF Relations

### Rule
Every LSIF edge (any triple whose predicate falls under the `lsif:` namespace URI) must connect a subject and an object node that are actively defined within the snapshot graph. This ensures there are no broken links in the compilation index.

### SPARQL 1.1/1.2 ASK Query
```sparql
PREFIX lsif: <https://microsoft.github.io/language-server-protocol/specifications/lsif/0.6.0/specification/>

ASK {
  ?s ?p ?o .
  FILTER(STRSTARTS(STR(?p), "https://microsoft.github.io/language-server-protocol/specifications/lsif/0.6.0/specification/"))
  FILTER(isIRI(?o) || isBlank(?o))
  FILTER NOT EXISTS { ?o ?any_p ?any_o }
}
```

### Interpretation
- **ASK Evaluation**: The query returns `true` if any triple has an `lsif:` predicate but its object `?o` (restricted to IRIs or blank nodes) does not appear as a subject of any statement (i.e. it is an orphan).
- **Control Plane Invariant Assertion**: The database state is considered **valid** only if this query returns `false`.

---

## 3. Invariant 2 — No Unreceipted Graph Consequence

### Rule
All diagnostic reports, compiled capabilities, and tool projections generated by the control plane must be traceable to a specific transaction and cryptographic receipt. Unreceipted outputs violate the auditability mandate.

### SPARQL 1.1/1.2 ASK Query
```sparql
PREFIX prov: <http://www.w3.org/ns/prov#>
PREFIX max:  <urn:lsp-max:core:>

ASK {
  ?artifact a max:Artifact .
  FILTER NOT EXISTS {
    ?artifact prov:wasGeneratedBy ?receipt .
    ?receipt a max:Receipt .
  }
}
```

### Interpretation
- **ASK Evaluation**: The query returns `true` if there is any instance of class `max:Artifact` that is not linked to a `max:Receipt` via the `prov:wasGeneratedBy` property.
- **Control Plane Invariant Assertion**: The database state is considered **valid** only if this query returns `false`.

---

## 4. Invariant 3 — No Hot-Path SPARQL Dependency

### Rule
Evaluating interactive LSP client definition/hover queries must never trigger synchronous database lookups or run SPARQL queries against Oxigraph. Instead, all ranges mapped to definition results must have pre-computed projections in the materialized view store.

### SPARQL 1.1/1.2 ASK Query
```sparql
PREFIX lsif: <https://microsoft.github.io/language-server-protocol/specifications/lsif/0.6.0/specification/>
PREFIX max:  <urn:lsp-max:core:>

ASK {
  ?range a lsif:Range .
  ?range lsif:textDocument_definition ?defResult .
  FILTER NOT EXISTS {
    ?projection a max:Projection ;
                max:sourceRange ?range .
  }
}
```

### Interpretation
- **ASK Evaluation**: The query returns `true` if any `lsif:Range` containing a `textDocument_definition` relationship lacks a corresponding pre-materialized `max:Projection` node.
- **Control Plane Invariant Assertion**: The database state is considered **valid** only if this query returns `false` (meaning the materialized views are complete).

## 5. Invariant 4 — No Ontology Laundering

### Rule
Custom tool extensions, proprietary metadata, and user-defined variables are strictly forbidden from using the official Microsoft LSIF namespace prefix. Custom properties must be isolated under the private `max:` prefix.

- **Control Plane Invariant Assertion**: The database state is considered **valid** only if this query returns `false`.

## 6. Invariant 5 — No False ALIVE

### Rule
The promotion of a release candidate to the ALIVE state is blocked if independent query executions yield results that differ from the values stored in the cryptographic receipt. This enforces complete replay determinism.

### SPARQL 1.1/1.2 SELECT Query
```sparql
PREFIX max:  <urn:lsp-max:core:>

SELECT ?receipt ?expectedResultHash ?actualResultHash WHERE {
  ?receipt a max:Receipt ;
           max:resultHash ?expectedResultHash ;
           max:queryHash ?qHash ;
           max:graphHash ?gHash .
  
  ?replay a max:Replay ;
          max:queryHash ?qHash ;
          max:graphHash ?gHash ;
          max:resultHash ?actualResultHash .
          
  FILTER(?expectedResultHash != ?actualResultHash)
}
```

### Interpretation
- **SELECT Evaluation**: The query searches for pairs of receipts and independent replays executed against the same query and input graph state. It matches any pairs where the actual result hash differs from the expected hash.
- **Control Plane Invariant Assertion**: The release candidate is promoted to ALIVE only if this query yields **0 solutions** (i.e. the result set is completely empty).
